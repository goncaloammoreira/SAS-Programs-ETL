# 🛒 SAS ETL Pipeline - Aperam Sourcing

This repository contains the SAS script responsible for the **Extraction, Transformation, and Loading (ETL)** process that consolidates SAP document flows to generate the Sourcing Spend analytical base (`sap_spend`).

---

## 🚀 Process Overview

The script acts as a data integrator for SAP ERP data hosted on **SAP Datasphere / HANA**. It maps the entire lifecycle from the Sales Document (SO) to the Purchasing Document (PO), cross-referencing this data with Purchase Order History (GR/IR) and accounting entries (BSEG/BKPF).


```

[SAP HANA (Datasphere)]
│
├──> Transposition/Concatenation (ZTEKPO) ──> [ztekpo_final]
│                                                 │
└──> Main Join (VBFA + EKPO + EKKO + EKBE...) ◄───┘
│
▼
[U:\AperamDWH\Sourcing\Data_output\sap_spend.sas7bdat]

```

---

## 📋 Environment Configuration & Connections

The script utilizes two main libraries to manage data traffic:

* **`hana` (Source):** An ODBC connection targeting the `SAPDS_DWH` schema in SAP Datasphere.
* **`sourcing` (Target):** Aperam's local network directory used to store the final Data Mart:
    * Path: `U:\AperamDWH\Sourcing\Data_output`

> ⚠️ **Security Note:** Access credentials for HANA are currently hardcoded at the beginning of the script. It is highly recommended to migrate these to secure global environment variables or integrated authentication in future production releases.

## ⚙️ Script Architecture & Modules

The ETL pipeline is divided into three sequential steps executed in batch mode:

### 1. Helper Macros
* `%formatCol`: Cleans and converts numeric ID columns into standardized 10-character string formats.
* `%formatDat`: A utility macro that converts date strings into the standard SAS date format (`yymmdd8.`).

### 2. Chemical Component Aggregation (`ZTEKPO`)
Alloy and chemical component tables often contain multiple rows ($N$ rows) for a single purchase order item. The script executes a `DATA step` utilizing a `RETAIN` statement to transpose these multiple rows into a single, comma-separated string grouped by `EBELN` (PO Number) and `EBELP` (PO Item).
* **Transposed Fields:** Chemical components, base price, exchange rates, and laboratory analysis fields (`f_analysis1` to `3`).

### 3. Spend Base Consolidation (`PROC SQL`)
The core of the script runs a massive join using the commercial document flow table (`VBFA`) as the main driver (`VBTYP_N = 'V'` and `VBTYP_V = 'C'`).

## 🧠 Core Business Rules Implemented

### A. Entity Separation (Sourcing vs. Mill)
The script conditionally splits metrics and codes depending on the Company Code (`BUKRS`):
* **Sourcing (`10H2`):** Captures vendor master data (`LFA1`), net prices, payment terms, and sourcing-specific timeline dates.
* **Mill (Plants/Usinas):** Captures the manufacturing side, including mill codes (`s_millCode`), plant codes (`WERKS`), and storage locations (`LGORT`).

### B. Purchase History & GR/IR Metrics Calculation
Using the Purchasing Document History table (`EKBE`) and the history type (`VGABE`), the script calculates financial volumes while accounting for Debit/Credit indicators (`SHKZG` where `S` = Debit/Plus, `H` = Credit/Minus):
* **`VGABE = '1'`**: Total Goods Receipt (GR) Amount/Quantity.
* **`VGABE = '2'`**: Total Invoice Receipt (IR) Amount/Quantity.
* **`VGABE = '3'`**: Total Debit/Credit Note Amount/Quantity.
* **`f_calculatedPoHistoryFinalAmount`**: Automated business rule. If the Invoice Receipt (IR) amount is zero, it defaults to the Goods Receipt (GR) amount. Otherwise, it consolidates the sum of IR + Debit/Credit Notes.

### C. Performance & Scope Filters
* **Cut-off Date:** The `HAVING` clause restricts processing to records where the sourcing purchasing document date (`BLDAT` for `10H2`) is strictly greater than **December 31, 2015**.
* **Deletion Indicator:** Items marked for deletion in SAP (`ekpo.loekz`) are automatically filtered out.

## 📊 Output Table Schema (`sourcing.sap_spend`)

Below are some of the key columns structured in the final output dataset:

| SAS Field Name | SAP Source Table | Description |
| :--- | :--- | :--- |
| `s_soNum` | `VBFA.VBELV` | Sales Document Number (Original Sales Order) |
| `s_poNumSourcing` | `VBFA.VBELN` | Purchase Order Number generated by Sourcing (`10H2`) |
| `s_poNumMill` | `VBFA.VBELN` | Purchase Order Number generated by the Mill |
| `s_supplierName` | `LFA1.NAME1` | Vendor / Supplier Name |
| `f_calculatedPoHistoryFinalAmount` | *Calculated* | Final consolidated financial Spend value |
| `s_zChemicalComponent` | `ZTEKPO` | Concatenated list of chemical components |
| `d_clearingDate` | `BSEG.AUGDT` | Accounting clearing date for the invoice |

## 🧹 Memory Cleanup & Execution

At the end of the run, the script invokes `PROC DATASETS` with a `DELETE` statement to purge the temporary datasets (`ztekpo_concat` and `ztekpo_final`) from the `WORK` library. This ensures optimal disk space and prevents memory overflow on the SAS server.

To run the pipeline, ensure you have active access to the Aperam network drive and execute the entire script (`F5` in SAS Enterprise Guide).
